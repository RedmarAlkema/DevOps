# Wat Jij Zelf Moet Regelen

## Waarom deze lijst handig is

Er zijn dingen waar ik je goed bij kan helpen, maar die ik niet volledig voor je kan afronden omdat jij daar zelf toegang, accounts, keuzes of toestemming voor nodig hebt.

Als jij deze punten regelt, kan ik daarna veel makkelijker de rest voor je uitwerken.

## 1. Nieuwe Git-repository en platformkeuze

Jij moet zelf bepalen:

- waar de nieuwe repository komt
- of je `GitHub`, `GitLab` of iets anders gebruikt
- of de repository `public` of `private` wordt
- wie toegang krijgt

Ik kan daarna wel helpen met:

- repo-structuur
- `README`
- `.gitignore`
- commitstrategie
- CI-bestanden

## 2. Inloggegevens en accounts

Jij moet zelf regelen:

- MongoDB toegang
- RabbitMQ toegang
- Gmail of SMTP-account
- Imagga-account of API-keys
- eventueel cloudplatform-accounts

Ik kan en moet die accounts niet namens jou aanmaken of beheren.

## 3. Echte secrets en wachtwoorden

Jij moet zelf invullen:

- `.env` waarden
- `JWT_SECRET`
- `GATEWAY_KEY`
- `MONGO_URI`
- `RABBITMQ_URL`
- `EMAIL_USER`
- `EMAIL_PASS`
- Imagga-keys

Ik kan wel aangeven:

- welke variabelen je nodig hebt
- hoe je ze veilig opslaat
- hoe je voorbeeldbestanden zoals `.env.example` maakt

## 4. Schoolopdracht-eisen en beoordelingscriteria

Jij moet zelf controleren:

- wat je docent precies verwacht
- of Docker verplicht is
- of CI/CD verplicht is
- of een cloud-deployment verplicht is
- of screenshots, filmpje of verslag verplicht zijn

Ik kan daarna wel helpen om alles precies op die eisen af te stemmen.

## 5. Keuzes die inhoudelijk van jou moeten komen

Jij moet zelf beslissen:

- wat de scope van je opdracht wordt
- welke DevOps-verbeteringen je echt wilt toevoegen
- of je alleen lokaal demonstreert of ook deployed
- of je bugs eerst fixt of vooral documentatie en infrastructuur doet

Ik kan daarna wel een concreet plan uitwerken op basis van jouw keuze.

## 6. Installatie van externe software

Ik kan je vertellen wat je nodig hebt, maar jij moet zelf zorgen dat het op jouw machine beschikbaar is:

- Node.js
- MongoDB
- RabbitMQ
- Git
- Postman
- Docker Desktop, als je dat wilt gebruiken

## 7. Externe portals en dashboards

Jij moet zelf inloggen in:

- GitHub of GitLab
- MongoDB Compass of Atlas
- RabbitMQ Management
- Gmail of je SMTP-provider
- Imagga dashboard

Ik kan die dashboards niet zelfstandig namens jou beheren zoals jij dat als eigenaar doet.

## 8. Definitieve beslissingen over publicatie

Jij moet zelf bepalen:

- welke bestanden wel of niet mee de repository in gaan
- of testdata mag blijven staan
- of documentatie namen goed genoeg zijn voor school
- of gevoelige gegevens echt verwijderd zijn

Ik kan dit wel met je nalopen en er advies over geven.

## 9. Demo en presentatie

Jij moet zelf regelen:

- wanneer je demo geeft
- welke flow je wilt laten zien
- of je live internet hebt tijdens de demo
- of je docent toegang moet krijgen tot de repo

Ik kan daarna wel helpen met:

- demo-script
- presentatievolgorde
- spreektekst
- testchecklist

## 10. Eventuele deployment-accounts

Als je dit verder DevOps-proof wilt maken met deployment, dan moet jij zelf accounts en toegang regelen voor bijvoorbeeld:

- GitHub Actions secrets
- Docker Hub
- Render
- Railway
- Vercel
- Azure
- AWS

Ik kan daarna wel configuratiebestanden en stappen voor je maken.

## 11. Wat jij het beste eerst kunt doen

Als je wilt dat ik daarna zo veel mogelijk voor je kan maken, regel dan eerst deze dingen:

1. bepaal je schoolscope
2. kies waar je nieuwe Git-repo komt
3. zorg dat je lokale omgeving stabiel draait
4. verzamel alle benodigde secrets en accounts
5. beslis of je Docker en CI/CD wilt meenemen

## 12. Wat ik daarna waarschijnlijk wél grotendeels voor je kan maken

Als jij bovenstaande regelt, kan ik daarna waarschijnlijk voor je uitwerken:

- hoofd-`README`
- `.gitignore`
- `.env.example`
- DevOps-documentatie
- health-check routes
- Dockerfiles
- `docker-compose.yml`
- opstartinstructies
- testplan
- demo-script
- CI-pipeline basis
- opschoningsplan voor de repo

## Korte samenvatting

Wat jij vooral zelf moet regelen:

- accounts
- wachtwoorden en keys
- repo-keuzes
- schoolvereisten
- externe installaties
- definitieve publicatiebeslissingen

Als dat staat, kan ik je daarna juist heel veel concrete projectstukken uit handen nemen.
